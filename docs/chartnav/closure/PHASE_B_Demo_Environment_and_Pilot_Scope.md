# Phase B — Demo Environment and Pilot Scope

## 1. Problem solved
ChartNav demos today require a developer to hand-seed a local instance before every call, and pilot conversations stall at "what exactly is the scope — how many providers, how long, what does success look like?" Both problems compound: an inconsistent demo surface produces inconsistent follow-up questions, and a loose pilot scope produces loose exit conversations that never close. This spec delivers a deterministic, dockerized demo environment and a signed-ahead-of-time pilot scope template with measurable exit criteria.

## 2. Current state (honest)
- The backend runs under uvicorn locally; there is a `docker-compose.yml` for Postgres but no single-command "bring up everything including seeded data" flow.
- Test factories exist in `backend/tests/factories.py` but produce random data intended for test isolation, not demo continuity.
- The Playwright video-clip pack lives under `artifacts/video_clips/` with four clips (login, encounter-create, transcript-to-soap, sign-flow). It is not refreshed by CI.
- No `scripts/seed_demo.py` exists. `ls backend/scripts/` confirms.
- No pilot scope template exists under `docs/pilot/`.
- Existing tests: 231 backend (`pytest backend/`) + 55 frontend (`npm test` in `frontend/`). Axe-AA gate present. These must continue to pass after demo seeding lands.

## 3. Required state
- `docker compose -f docker-compose.demo.yml up` brings up Postgres, backend, frontend, and a seeded database in under 3 minutes on a recent laptop.
- Deterministic seed: same inputs, same output. Running twice produces the same IDs where possible (UUIDv5 off fixed namespaces) so screenshots remain stable across demos.
- Seed content:
  - 3 clinician identities (2 general ophthalmology, 1 retina) + 1 admin + 1 front_desk + 1 reviewer.
  - 20 encounters distributed across the four Phase A specialty templates (general, glaucoma, cataract, retina).
  - 1 signed encounter with a linked post-visit summary and a consult letter (demonstrates the full end-to-end).
  - Reminders spanning 14 days with 3 overdue, 5 complete, 4 upcoming.
  - Intake tokens: 2 pending, 1 accepted, 1 expired.
- Playwright clip pack is regenerated by a single command and stored in `artifacts/video_clips/` as deterministic MP4s.
- Pilot scope template is a filled-in example at `docs/pilot/scope-template.md` — a template, not a placeholder — with sample numbers and exit criteria, ready to be duplicated and edited per pilot.

## 4. Acceptance criteria (testable)
- `backend/tests/test_seed_demo.py`:
  - Running `scripts/seed_demo.py` against an empty DB produces exactly the documented row counts across tables.
  - Running twice is idempotent; no duplicate rows, no conflicts.
  - All seeded records are scoped to a single demo organization; no other orgs exist after seeding.
- Frontend smoke: `e2e/demo_smoke.spec.ts` logs in as each seeded identity and asserts the expected role-gated surfaces render without console errors.
- CI artifact: the clip pack is regenerated on every merge to `main`; missing or stale clips fail the CI job.
- Pilot scope template completeness check: `docs/pilot/scope-template.md` fills every section of the schema below — no TBDs.

## 5. Codex implementation scope
Create:
- `docker-compose.demo.yml` — compose file with depends_on wiring, healthcheck-gated seed step.
- `backend/scripts/seed_demo.py` — idempotent seeding; uses UUIDv5 from a fixed namespace string.
- `backend/scripts/seed_data/` — JSON fixtures (clinicians, encounters, reminders, intakes).
- `backend/tests/test_seed_demo.py`.
- `frontend/e2e/demo_smoke.spec.ts`.
- `docs/pilot/scope-template.md` — pilot scope template (see layout below).
- `docs/pilot/exit-criteria.md` — rationale document for the metrics.
- `Makefile` targets (or `justfile`): `demo-up`, `demo-seed`, `demo-clips`.

Modify:
- `artifacts/video_clips/README.md` — document the regeneration command and retention.
- `.github/workflows/ci.yml` — add a `demo-clips` job on `main` merges.

Pilot scope template layout (`docs/pilot/scope-template.md`):
```markdown
## Parties
- Practice name, primary contact, clinical lead
- ARCG contact, support path

## Scope
- Deployment mode (standalone | integrated_readthrough | integrated_writethrough)
- Providers in pilot (1-3)
- Specialty templates enabled (general, glaucoma, cataract, retina)
- Pilot duration: 30/60/90 days

## Exit criteria (measurable)
- Median time from transcript ingest to signed note: target N minutes
- Missing-flag resolution rate: target X%
- Reminder completion rate: target Y%
- Clinician NPS (small N, collected honestly): target score, acknowledge sample size

## Support
- Business-hours email: support@arcg...
- Optional weekly sync: day/time
- Escalation path: Sev-1 / Sev-2 / Sev-3

## Data and privacy
- BAA signed: yes/no
- Data residency: region
- Data handling at pilot end: retain | purge | export
```

## 6. Out of scope / process only
- Production-grade multi-tenant seed; demo data is a single-org illustration.
- Realistic transcript audio; seeded encounters carry pre-generated transcripts, not actual STT runs.
- Automated demo-data teardown per tenant; operators run `docker compose down -v` manually.
- Signed pilot MSA templates. The scope template is operational, not contractual.

## 7. Demoable now vs later
- Demoable on ship: a one-command demo bring-up, a full-journey signed-encounter path, and a filled pilot scope document as a leave-behind after the call.
- Not demoable: a live pilot dashboard aggregating multiple pilots at once (single-org today).
- Not demoable: production data migration from a buyer's EHR into the demo environment — that's a pilot kickoff task, not a demo task.

## 8. Dependencies
- Phase A ophthalmology templates must be final; the seed encounters reference template IDs.
- Referring-Provider, Post-Visit Summary, and Messaging specs must be at least partially landed so that the full-journey encounter can render a summary and a consult letter.
- Admin Dashboard must be live so the seeded data produces non-empty trend lines.

## 9. Truth limitations
- The demo environment is not HIPAA-compliant; it uses synthetic data but runs on a developer laptop. We never demo against real patient records.
- Numbers in the demo dashboard are seeded, not real. The pilot scope template explicitly re-iterates that live pilot metrics will differ.
- The clip pack shows the product as it exists today; when product changes, stale clips linger until the CI regenerates them.

## 10. Risks if incomplete
- Inconsistent demos are the single largest source of qualification drift — buyers end up evaluating different products than the one we plan to ship.
- Unsigned exit criteria at pilot start lead to unconverted pilots: 60–90 days of work and no clear "did we pass." Every ARCG pilot conversation should start with an exit-criteria doc on the table.
- Without the scope template, sales engineering re-invents scope on every call; the pilot-conversion ratio drops because scope ambiguity reads as immaturity.
